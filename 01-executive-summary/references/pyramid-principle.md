# The Minto Pyramid Principle — Practitioner Reference

## What This Is

Barbara Minto developed the Pyramid Principle at McKinsey & Company in the 1960s. It became the de facto standard for consulting report writing. The core insight: *readers comprehend ideas presented from the top down, in a pyramidal structure*. Unclear writing deviates from that structure. Clear writing obeys it.

The framework applies to any document where the purpose is to present thinking clearly — executive summaries, proposals, funding requests, reports, memos.

---

## 1. The Pyramid Structure

### The Core Claim

> "Controlling the sequence in which you present your ideas is the single most important act necessary to clear writing. The clearest sequence is always to give the summarising idea before you give the individual ideas being summarised." (Minto, 2010, p.5)

### Structure Diagram

```
                    ┌─────────────────────────┐
                    │  GOVERNING THOUGHT       │   ← Top Box: answers the reader's question
                    │  (Main Point / Answer)   │
                    └───────────┬─────────────┘
                                │ raises new question
                    ┌───────────┴─────────────┐
                    │      KEY LINE           │   ← 3-5 points that answer it
              ┌─────┴──────┬────────┬─────────┤
              ▼            ▼        ▼         ▼
         Key point 1  Key point 2  Key point 3 ...
              │            │        │
         Supporting    Supporting  Supporting
         detail         detail     detail
```

### Three Rules (Must All Be Obeyed)

| Rule | Statement | Why It Matters |
|------|-----------|----------------|
| **Rule 1** | Ideas at any level must always be *summaries* of the ideas grouped below them | Forces abstraction; prevents lists masquerading as arguments |
| **Rule 2** | Ideas in each grouping must always be *the same kind of idea* | Ensures proper classification — all reasons, all steps, all problems |
| **Rule 3** | Ideas in each grouping must always be *logically ordered* | Reveals missing points; proves the grouping is valid |

If any rule is broken, there is a flaw in the thinking — not just the writing.

### The Vertical Relationship: Question/Answer Dialogue

Every statement raises a logical question in the reader's mind. The writer must answer it at the level below, before raising another. The chain continues until the reader has no more logical questions.

- State the governing thought → reader asks "Why?" or "How?"
- Answer at Key Line level → reader asks "How does that work?"
- Answer at support level → reader is satisfied

> "The pyramid structure almost magically forces you to present information only as the reader needs it." (Minto, 2010, p.14)

### The Horizontal Relationship: Deductive or Inductive

Ideas on the same level must form either a deductive or an inductive argument — never a mixture.

## 2. SCQA — The Opening Formula

The introduction to any business document should follow the narrative structure:

**Situation → Complication → Question → Answer**

### What Each Element Does

| Element | Definition | Purpose |
|---------|-----------|---------|
| **Situation** | A statement about the subject the reader will accept as true, anchoring them in a specific time and place | Establishes shared ground; reader nods and says "Yes, I know that — so what?" |
| **Complication** | Something that happened in the Situation that created a tension, problem, or need for action | Triggers the Question naturally |
| **Question** | The question the reader would logically ask, given the Situation + Complication | Defines what the document must answer |
| **Answer** | The governing thought — the direct answer to the Question | The top box of your pyramid |

### The Bare-Bones Formula

```
S = We have/do X (stable situation)
C = Something changed / went wrong / was discovered (complication)
Q = So what should we do? / Is this the right approach? / How do we do it?
A = [Your recommendation / conclusion]
```

### Four Order Variations

The S-C-Q elements can be arranged in different sequences to set different tones:

1. **Standard (S → C → A)** — Most common. Neutral, professional tone.
2. **Direct (A → S → C)** — Lead with the recommendation. Used when reader wants the bottom line first.
3. **Concerned (C → S → A)** — Lead with the problem. Used when the complication is urgent.
4. **Question first (Q → S → C → A)** — Used in directive memos or when the question is already in the reader's mind.

### Common Business Document Patterns

| Document Type | S | C | Q |
|--------------|---|---|---|
| Directive / memo | We want to do X | Need you to do Y | How do you do Y? |
| Funding request | We have a problem | We have a solution costing $X | Should you approve? |
| Progress review | We agreed to do X | We have now done it | What did we find? |
| Consulting proposal | Client has a problem (R1, R2) | They asked us to investigate | What should they do? |

### Key Rule for Introductions

> "Introductions are meant to *remind* rather than to *inform*. Nothing should be included that would have to be proved to the reader for him to accept the statement of your points." (Minto, 2010, p.47)

The introduction contains only what the reader already knows or could accept as true — no exhibits, no new data. Historical chronology belongs in the introduction, not in the body.

## 3. Deductive vs. Inductive Reasoning

### Deductive Reasoning

A line of argument leading to a "therefore" conclusion. Each point derives from the one before.

**Structure:**
```
Statement about a situation in the world
+ A second statement commenting on its subject or predicate
= An implication of both existing simultaneously ("therefore")
```

**Classic example:**
- Men are mortal.
- Socrates is a man.
- Therefore Socrates is mortal.

**Summary of a deductive grouping** rests heavily on the final conclusion:
"Because Socrates is a man, he is mortal."

**Limit:** No more than four points in a deductive argument; no more than two chained "therefore" steps.

### Inductive Reasoning

A group of ideas that are the same *kind* of thing, from which you draw a general inference about their sameness.

**Structure:**
```
[Idea A] [Idea B] [Idea C] ← all describable by the same plural noun
→ Inference about what their similarity implies
```

**Classic example:**
- French tanks are at the Polish border.
- German tanks are at the Polish border.
- Russian tanks are at the Polish border.
- → Poland is about to be invaded by tanks.

**Summary of an inductive grouping** states the inference drawn from the similarity, not just the category label.

### When to Use Which

| Situation | Recommended Form |
|-----------|-----------------|
| Key Line level (main document structure) | **Inductive** — easier for reader to absorb |
| Paragraph level | **Deductive** — short enough to follow |
| Reader expects an unexpected recommendation | **Deductive** — argument must come before conclusion |
| Reader needs to understand method before steps | **Deductive** — e.g., explaining a new analytical technique |
| All other cases | **Inductive** — state the action/recommendation, then support it |

> "Always present action before argument, since that is what the reader cares about — unless the argument is what he really cares about." (Minto, 2010, p.65)

### Telling Them Apart

- If the **second point comments on the subject or predicate of the first**, it is deductive.
- If the **second point adds another member of the same class**, it is inductive.
- If neither applies, the points do not belong together.

## 4. MECE — Grouping Rules

**MECE** = Mutually Exclusive, Collectively Exhaustive

Defined by Minto as the test for any valid grouping:

- **Mutually exclusive**: no overlap between items in the group (what goes into Box A does not also belong in Box B)
- **Collectively exhaustive**: the group contains everything relevant — nothing left out

> "When you divide a whole into its parts, you must make sure that the pieces you produce are mutually exclusive of each other and collectively exhaustive in terms of the whole." (Minto, 2010, p.82)

### Practical Test

Can you describe all the items in the grouping with a single **plural noun**?
- Steps ✓ | Reasons ✓ | Problems ✓ | Recommendations ✓ | Risks ✓
- If you cannot find a plural noun that fits, the grouping is invalid.

### The Three Logical Orders

Every valid grouping of inductive ideas must have one of three orders, reflecting the analytical activity that created it:

| Order | Source | When to Use |
|-------|--------|-------------|
| **Time order** | Actions in a process (cause → effect) | Steps in a plan; implementation phases; process description |
| **Structural order** | Parts of a visualised whole (geography, org chart, value chain) | Market segmentation; organisational analysis; industry structure |
| **Degree order** | Classification ranked by a shared characteristic | Problems ranked by severity; markets ranked by size; risks ranked by likelihood |

**If no order is present, the grouping is invalid.**

## 5. The "So What?" Test — Finding the Governing Thought

### The Test

After reading any statement you have written, ask: *"So what? Why are you telling me this?"*

If you cannot answer that question clearly, the point either:
- Does not belong in the document
- Belongs in the introduction (as Situation context), not the body
- Is missing its summary — you have listed data without drawing the insight

### Intellectually Blank Assertions

The most common error. These are summary statements that name the *kind* of idea rather than stating *what* the ideas mean:

| Blank Assertion | What to Replace It With |
|----------------|------------------------|
| "The company has three problems." | State what the problems collectively mean: "The company cannot scale because it has delegated no authority." |
| "We have five recommendations." | State the net effect: "We can cut composing costs by 30% by simplifying methods and raising wages." |
| "There are four variables to manage." | State why managing them matters: "These four variables together determine whether the resource allocation process produces the right output." |

> "Intellectually blank assertions are deadly for the reader because they do not anchor his mind, they are not stimulating to read, and they present the very real danger that he will not grasp what you are trying to say." (Minto, 2010, p.94)

### How to Find the Real Summary

For **action ideas** (steps, recommendations): state the *effect* of carrying out the actions.
For **situation ideas** (reasons, problems, conclusions): state *what their similarity implies*.

Both demand completing the thinking — not just listing.

## 6. Common Structural Errors

### Error 1: The Intellectually Blank Assertion
*Symptom:* Summary says "three reasons" or "five problems" rather than what those reasons mean.
*Fix:* Ask "If all these points are true, what does that tell us?" State that insight.

### Error 2: False Grouping (Violates MECE)
*Symptom:* Ideas in a group cannot be described by a single plural noun; some overlap; some belong elsewhere.
*Fix:* Identify the real classification. Sort items into proper sub-groups. Find the underlying order (time / structural / degree).

### Error 3: Mixed Reasoning Levels
*Symptom:* A group contains both causes and effects treated as equals (e.g., "low productivity" and "high overtime" and "uncompetitive prices" listed as three separate problems).
*Fix:* Recognise the cause-effect chain. The true structure is deductive: low productivity → high overtime → uncompetitive prices. State the governing conclusion.

### Error 4: Deductive Argument at the Key Line
*Symptom:* The executive summary leads with "Here is what is going wrong / Here is why / Here is what to do." Reader must hold A1, B1, C1 in mind while ploughing through 10 pages before reaching A3, B3, C3.
*Fix:* Rotate 90 degrees. Put the recommendations on the Key Line; group the supporting findings underneath each one.

### Error 5: News Instead of Thinking
*Symptom:* A grouping contains true facts that share no logical relationship — they cannot be described by a plural noun because there is nothing the same about them.
*Fix:* Remove them. "The only justification for including a point in a document is that, together with others, it helps to explain or defend a higher point." (Minto, 2010, p.71)

### Error 6: Historical Chronology in the Body
*Symptom:* The body of the document recounts what happened in sequence, without analysis.
*Fix:* Move events to the Situation/Complication in the introduction. The body contains only ideas (statements that raise questions) in logical order.

### Error 7: Vague Action Wording
*Symptom:* Steps are worded as activities rather than end products: "Strengthen regional effectiveness" instead of "Assign planning responsibility to the regions."
*Fix:* Visualise the end product the reader will have in hand after completing the action. Word the step to describe that end product.

## 7. Building a Pyramid — Step-by-Step

### Top-Down Approach (Preferred)

Use when you know your subject and can identify the reader's question.

1. **Draw the top box.** Write the subject.
2. **Identify the Question.** What does the reader need answered about the subject?
3. **State the Answer.** This is your governing thought / main point.
4. **Validate via Situation.** What is the first noncontroversial thing you can say about the subject that the reader will accept as true?
5. **Develop the Complication.** What happened in that Situation to raise the Question?
6. **Recheck Q and A.** Does the Complication naturally lead to the Question? Does the Answer directly address it?
7. **Find the Key Line.** What new question does the Answer raise? What plural noun labels the answer to that question (reasons? steps? changes?)?
8. **Support each Key Line point** with the same Q/A process, level by level.

### Bottom-Up Approach (When Top is Unclear)

Use when you have a pile of data points but cannot yet see the structure.

1. **List all the points** you think you want to make.
2. **Work out the relationships** — draw cause-effect chains, look for groupings by type.
3. **Draw conclusions** from each grouping.
4. **Think through the introduction** to find the governing question and work backwards to confirm the top point.

### The 30-Second Test

> "If your entire thinking is not clear to the reader in the first 30 seconds of reading, you should rewrite." (Minto, 2010, p.29)

After writing, check: can a reader get the full message — governing thought + Key Line points — in the opening paragraph? If not, restructure.

## 8. Problem-Definition Framework

For complex business documents (reports, proposals, consulting outputs), the SCQA introduction maps to a formal problem-definition:

```
Situation = Starting Point/Opening Scene + Disturbing Event
R1        = Undesired Result (the problem as it currently stands)
R2        = Desired Result (what the reader wants instead)
Question  = How do we get from R1 to R2?
Answer    = [Your recommendation / solution]
```

### The Five Elements

| Element | Description |
|---------|-------------|
| **Starting Point / Opening Scene** | The stable structure or process the reader recognises. "Once upon a time there was a company that..." Anchor in a specific time and place. |
| **Disturbing Event** | What changed — external (new competitor, technology shift, policy change) or internal (new system installed, market expansion, lagging performance revealed). |
| **R1 (Undesired Result)** | What the Disturbing Event produced that the reader does not want. State briefly: declining sales, eroding margins, inability to serve the market. |
| **R2 (Desired Result)** | What the reader wants instead. State specifically and measurably: "Reduce time to market by one-third." Without a clear R2, you cannot choose between solutions. |
| **Question** | Determined by where the reader stands in solving the problem (see table below). |

### Seven Common Reader Situations

| # | Reader's Position | Question |
|---|------------------|----------|
| 1 | Does not know how to get from R1 to R2 | What should we do? |
| 2 | Has a suggested solution, unsure if right | Should we do it? |
| 3 | Knows the solution, unsure how to implement | How do we do it? |
| 4 | Implemented a solution that did not work | What should we do now? |
| 5 | Has two alternative solutions | Which should we choose? |
| 6 | Has already decided and is asking for a plan | How will you do it? |
| 7 | Wants to understand what happened | Why did it happen? |

## 9. Key Quotes for Business Plans

These can be cited parenthetically as (Minto, 2010) in executive summaries or methodology sections:

> "The clearest sequence is always to give the summarising idea before you give the individual ideas being summarised." (p.5)

> "The key to clear writing is to slot your ideas into this pyramidal form and test them against the rules before you begin to write." (p.11)

> "A good introduction does more than simply gain and hold the reader's interest. It influences his perceptions. The narrative flow lends a feeling of plausibility to the writer's particular interpretation of the situation." (p.58)

> "The only justification for including a point in a document is that, together with others, it helps to explain or defend a higher point." (p.71)

## 10. Business Plan Application

| Business Plan Section | Pyramid Principle Application |
|----------------------|------------------------------|
| **01 Executive Summary** | This *is* the pyramid in compressed form: governing thought (top box) + key line points. Apply the 30-second test. Use SCQA as the opening paragraph structure. |
| **02 Company Overview** | SCQA: S = what we do and where we are; C = the gap or opportunity; Q = why us / why now; A = our proposition. |
| **03 Products / Services** | Key Line = 3-4 reasons this product/service wins. Each supported by MECE groupings (features → benefits → evidence). |
| **04 Market Analysis** | Structural order: divide the market (segments = mutually exclusive + collectively exhaustive). Degree order for opportunity ranking. |
| **05 Target Market** | Inductive grouping: define the target by a shared characteristic; draw the inference about their unmet need. |
| **06 Competitive Analysis** | Problem-definition framework: R1 = current customer pain with existing solutions; R2 = what they want; your product closes the gap. |
| **07 Marketing & Sales** | Action ideas ordered as a process (time order). Summary = the effect achieved, not "our marketing plan has five components." |
| **08 Operations Plan** | Time-ordered steps. Distinguish levels of action (strategic objectives vs operational steps). State each step as an end product. |
| **09 Management Team** | Inductive grouping: team members described by a shared plural noun (e.g., "operators with hands-on sector experience"). Summary = what that combination enables. |
| **10 Financial Projections** | Deductive argument permissible here: assumptions → calculations → projections. Keep it at paragraph level; Key Line stays inductive (three financial milestones, not the deductive chain). |
| **11 Funding Request** | Funding-request pattern: S = we have a validated business; C = we need capital to scale; Q = should you fund us? Key Line = use of funds + expected return + exit or repayment path. |
| **12 Risk Analysis** | Degree order: rank risks by likelihood × impact. Each risk = problem-definition sub-unit (R1 = risk materialises; R2 = avoided; solution = mitigation). |
| **13 Implementation Timeline** | Time order throughout. Distinguish levels: phases (strategic) → milestones (operational) → tasks (tactical). Summary of each phase = its end product. |
| **15 Appendices** | No pyramid needed — appendices answer "Where is the evidence?" for claims made in the body. |

### Quick Reference: SCQA for a Uganda Business Plan

```
S = [Industry / sector] in Uganda is [size, growth rate, current state].
    [Company name] was established in [year] to [core purpose].

C = Despite [market opportunity], [specific gap or problem remains].
    OR: Having [achieved milestone], the business now faces [constraint / next challenge].

Q = How can [company name] [achieve objective] in [timeframe]?
    OR: Is this the right time / approach to [major decision]?

A = [Company name] can achieve [specific result] by [3 key actions / reasons],
    requiring [specific investment or approval].
```
