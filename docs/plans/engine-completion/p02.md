# Phase 02 - Orchestration and State Management

## Objective

Turn the repository from a collection of strong individual skills into a controlled production workflow that reliably moves from intake to final deliverable without contradiction, missing evidence, or broken sequencing.

## Why This Phase Is Critical

The repo's own recommendations already identify missing end-to-end orchestration as a top weakness. This is the central systems problem. A top-tier consultancy engine needs:

- a required workflow
- explicit handoffs between skills
- structured state for key facts
- consistency checks across outputs

## Core Workstreams

### 1. Design the master workflow

Create one mandatory orchestration sequence for serious engagements:

1. client intake
2. country context
3. primary research and market validation
4. section drafting
5. financial model and valuation
6. consulting synthesis
7. stress test and bankability/due-diligence review
8. appendices and evidence matrix
9. assembly
10. pitch or proposal packaging where needed

### 2. Define structured state

Create a lightweight engagement-state specification that captures:

- client identity
- audience type
- sector
- geography
- core assumptions
- TAM/SAM/SOM values
- revenue model
- pricing assumptions
- capex and opex assumptions
- funding ask
- milestones
- team claims
- key risks

This should live in a reusable format under `docs/` and be referenced by orchestration-heavy skills.

### 3. Build cross-section consistency gates

Introduce a standard reconciliation layer so the engine checks whether:

- market sizing supports revenue projections
- staffing supports operations claims
- use of funds supports implementation milestones
- risk mitigations correspond to real risks
- executive summary matches the full plan

### 4. Upgrade `00-client-intake` and `00-plan-assembly`

These should become true control points rather than light wrappers. They should:

- enforce required inputs
- route to the right output type
- call the right meta-skills
- stop incomplete engagements before final assembly

## Repository Changes to Make

- Create a master orchestration reference in `docs/`.
- Enhance `00-client-intake/SKILL.md`.
- Enhance `00-plan-assembly/SKILL.md`.
- Add an engagement state template and consistency checklist.

## Deliverables

- Master workflow map
- Engagement-state template
- Consistency matrix
- Intake upgrade
- Assembly upgrade

## Exit Criteria

Phase 02 is complete only when:

- every major output can be traced through one standard workflow
- key numbers and claims have a shared reference structure
- assembly is impossible without passing minimum quality gates
- the system reduces contradiction risk materially

## Risks if Skipped

- excellent sections that do not add up to a decision-ready document
- repeated rework because assumptions drift between skills
- inconsistent output quality across engagements
- founder dependence because only one person can reconcile the full story mentally

## Suggested Reading Material

- *Business Plans to Game Plans* by Jan B. King
- *The Checklist Manifesto* by Atul Gawande
- *Managing the Professional Service Firm* by David Maister
- *The Goal* by Eliyahu Goldratt

Buy and provide these at this stage because this phase is about workflow control, handoffs, constraints, and operating discipline.
