# Validation Experiments

**Source:** Bland & Osterwalder, *Testing Business Ideas* (Wiley/Strategyzer, 2020)

Validation experiments confirm direction with **stronger evidence** — real actions, real payments, real commitments. Use after discovery experiments have given you a clear direction. Validation experiments cost more time and money, but reduce risk significantly before full build.

**Rule:** Escalate fidelity progressively. Concierge → Single Feature MVP → Full Build. Never skip validation to go straight to build.

---

## Interaction Prototypes

### 1. Clickable Prototype

**Purpose:** A digital mockup with clickable zones that simulates the product experience. Higher fidelity than paper; tests whether the flow makes sense before any real development.

**Evidence type:** Task completion + qualitative feedback → Moderate

**When to use:** After paper prototype testing has given you a clear direction on product flow.

**Process:**
1. Design screens in a prototyping tool (Figma, InVision, or even PowerPoint with links)
2. Create "hot zones" — clickable areas that navigate between screens
3. Test internally first; check flow works
4. Recruit 10–15 target customers; run facilitated sessions (one interviews, one notes)
5. Observe where customers get confused, stuck, or excited — do not explain or defend

**Key metrics:** Task completion rate; time to complete key tasks; qualitative themes

**Tools:** Figma (free tier), Marvel App, InVision, or Adobe XD

**EA adaptation:** For non-digital products (physical goods, services), use a "Clickable Brochure" — a printed or PDF with hyperlinked sections — or a WhatsApp conversation flow simulation.

### 2. Single Feature MVP

**Purpose:** A fully functioning minimum viable product that delivers a single core feature — the one most important to test. Customers receive real value; you receive real feedback and potentially real revenue.

**Evidence type:** Customer satisfaction + payments → Strong

**When to use:** After discovery and lower-fidelity validation have confirmed a specific customer need and a specific solution concept.

**Cost:** Higher than discovery experiments — 1–3 weeks to build; several weeks to run.

**Process:**
1. Design the smallest version of the feature that solves the most important customer job
2. Test internally to confirm it works
3. Acquire customers — use channels confirmed in discovery experiments
4. Deliver; collect payment if applicable
5. Gather structured satisfaction feedback

**Key metrics:**
- Customer satisfaction (1–5 scale; aim for 4+)
- Number of purchases
- Cost to deliver (unit economics)
- Customer quotes on what they valued and what was missing

**EA adaptation:** A Single Feature MVP for a food business might be one product sold through one channel. For a service business, it might be a single service package delivered manually. For software, it might be a WhatsApp-based service before building an app.

---

### 3. Mash-Up

**Purpose:** A functioning MVP that pieces together existing technology or services to deliver customer value, without building anything from scratch. Tests whether the business concept works before investing in custom development.

**Evidence type:** Customer satisfaction + payments → Strong

**Process:**
1. Map the process flow needed to deliver customer value
2. Survey existing tools, platforms, or services that can do each step
3. Piece them together; test the combined experience internally
4. Deliver to customers; measure satisfaction and payments
5. Only build custom components if the mash-up experience failed because existing tools are insufficient

**Examples:**
- Combine WhatsApp ordering + Google Sheets inventory tracking + MTN MoMo payments before building an app
- Use existing logistics providers for delivery before investing in own fleet
- Use a shared commercial kitchen before building your own food processing facility

**Key metrics:** Same as Single Feature MVP — satisfaction, payments, unit economics

### 4. Concierge

**Purpose:** Deliver the product or service entirely manually — with people instead of technology. The customer knows a human is doing the work. Tests whether the value proposition resonates and whether delivery is feasible before automating.

**Evidence type:** Customer satisfaction + real delivery → Strong

**When to use:** Ideal for service businesses. Essential for testing before investing in automation or technology infrastructure.

**Process:**
1. Plan all steps required to deliver the service manually
2. Take orders (via WhatsApp, in person, or simple form)
3. Deliver manually; track every step and how long it takes
4. Collect payment at real price points
5. Gather feedback after delivery
6. Use cost and time data to inform automation investment decision

**Key metrics:**
- Customer satisfaction after manual delivery
- Time per delivery (informs staffing/automation cost)
- Revenue per delivery; cost per delivery (unit economics)
- Number of repeat purchases (retention signal)

**EA adaptation:** Most successful Ugandan service businesses start as concierge operations — the founder delivers everything personally before hiring. This is the right approach. Systematise what works, then hire; automate only what systematisation cannot handle.

**Classic example:** The first food delivery businesses in Kampala operated as WhatsApp-order + boda boda delivery concierge operations before building apps.

### 5. Life-Sized Prototype

**Purpose:** A physical, full-size model of a product — not functional but visually and spatially accurate. Used when physical presence is essential to test customer response (e.g., furniture, equipment, retail space layout).

**Evidence type:** Physical interaction + qualitative feedback → Moderate

**Process:** Build from cardboard, foam, or cheap materials. Place in the customer's real environment if possible. Observe how they interact with it; record quotes and body language.

**EA adaptation:** Use locally available materials (plywood, cardboard, hardboard). Local fabricators (carpentry workshops, metal fabricators in Katwe or Kalerwe) can build mockups cheaply.

## Call to Action

### 6. Simple Landing Page

**Purpose:** A single web page that communicates the value proposition and asks for a specific action (email sign-up, payment, or enquiry). Used to test demand at low cost before building a product.

**Evidence type:** Sign-ups → Moderate; payments → Strong

**Process:**
1. Build a simple page with: headline (value proposition), 3 key benefits, one clear call-to-action (CTA)
2. Drive traffic via: online ads, social media, WhatsApp broadcast, email
3. Measure: unique visitors, CTA conversion rate, time on page

**Key metrics:**
- Email sign-up conversion rate: 5–20% is typical
- Payment conversion rate: 1–5% of visitors is a strong signal

**Tools:** Carrd (free), WordPress, Shopify, or even a WhatsApp Business profile with a link

**EA adaptation:** A WhatsApp Business profile link shared via broadcast is the lowest-friction equivalent of a landing page for many Ugandan audiences. Measure how many people click, start a conversation, and complete a purchase.

---

### 7. Crowdfunding

**Purpose:** Raise money for a product before building it. Tests viability (will people actually pay?) and provides funding to build. Also tests marketing and messaging at scale.

**Evidence type:** Real payments → Very Strong (strongest available for B2C products)

**Process:**
1. Create a compelling campaign: video, description, reward tiers
2. Set a minimum funding goal (the amount needed to produce the first batch)
3. Drive traffic from existing audience and press
4. Track: number of backers, average pledge, most popular tier

**Key metrics:** Funding reached / goal (100%+ = strong validation); number of backers; backer demographics

**EA adaptation:** International crowdfunding platforms (Kickstarter, Indiegogo) are difficult to use from Uganda due to payment limitations. Consider: pre-order campaigns via WhatsApp/Facebook with mobile money payment; local alternatives (Pesapal, Flutterwave, Paystack) for collecting pre-orders. Document all pre-orders formally.

### 8. Split Test (A/B Test)

**Purpose:** Run two or more versions of a message, offer, or design simultaneously to a split audience. Determines which version performs better with statistical confidence.

**Evidence type:** Comparative behaviour data → Moderate-Strong (depends on sample size)

**When to use:** Once you have enough traffic or audience to split meaningfully (usually 100+ events per variant).

**Process:**
1. Isolate one variable to test (headline, price, image, CTA, offer)
2. Create exactly two versions — identical except for the one variable
3. Randomly assign audience to each version
4. Measure conversion rate on your key metric for each
5. Run until one version is statistically significant (typically 95% confidence)
6. Implement winner; test next variable

**EA adaptation:** Useful for optimising WhatsApp message copy, Facebook ad creatives, and price point testing. Even manual A/B testing (send version A to first 100 WhatsApp contacts, version B to next 100) generates useful directional data.

### 9. Presale

**Purpose:** Collect payment for a product or service before it exists or before it is ready. The strongest real-world validation available — customers commit real money.

**Evidence type:** Real payments → Very Strong

**Process:**
1. Define the offer clearly: what will they receive, when, at what price
2. Communicate honestly that the product is not yet available
3. Accept payment via mobile money, bank transfer, or Pesapal
4. Track: number of presales, average order value, customer source

**Key metrics:**
- Minimum viable presale quantity: define upfront (e.g., "We will produce the first batch if we receive 50 orders by [date]")
- Revenue collected vs. cost-to-produce first batch

**Risks:** Legal obligation to deliver or refund. Do not over-commit on timeline. Be conservative on delivery dates.

**EA adaptation:** Presales via WhatsApp broadcast are common for Ugandan SMEs ("Pre-order now, we deliver in 2 weeks"). Formalise with written confirmation and mobile money receipts.

### 10. Validation Survey

**Purpose:** Gather quantitative data from a larger sample to confirm findings from qualitative experiments. Stronger than discovery surveys because questions are more specific and testing a well-formed hypothesis.

**Evidence type:** Stated intent at scale → Moderate

**Key difference from discovery survey:** Validation surveys test specific claims, not general pain exploration. Questions should be: "Here is our offer at this price — how likely are you to buy?" not "What problems do you have?"

**Key metrics:**
- % who say they are "very likely to buy" (target 40%+ for B2C; lower for B2B)
- Acceptable price range (Van Westendorp price sensitivity analysis)
- Willingness to pay by segment

**Avoid:** Asking "Would you buy this?" — over-inflated by social desirability. Instead: "If we launched tomorrow at UGX X, what would you do?" and offer specific options.

## Simulation

### 11. Wizard of Oz

**Purpose:** Deliver the product or service using hidden human effort that simulates an automated system. Unlike Concierge, the customer believes the product is automated. Tests desirability and demand before building the real system.

**Evidence type:** Real behaviour (customers think it's real) → Strong

**Process:**
1. Create a front-end interface that looks automated (landing page, app shell, WhatsApp bot interface)
2. Behind the scenes, humans manually fulfil every order or response
3. Customers interact as if with a real product
4. Measure: conversion, satisfaction, retention, willingness to pay

**Key decision point:** If Wizard of Oz generates strong positive evidence, the investment in building the real system is justified. If demand is weak, the Wizard reveals this before automation costs are incurred.

**EA adaptation:** WhatsApp "bots" operated manually are a Wizard of Oz in common Ugandan use. Many successful businesses run entirely as WhatsApp manual operations before building apps — this is valid validation.

**Classic example:** Airbnb's founders manually visited early hosts, photographed their properties, and responded to enquiries before building any of the platform infrastructure.

### 12. Mock Sale

**Purpose:** Present a real offer to customers and measure how many would pay — without actually charging them. Stronger than a survey; weaker than a presale.

**Evidence type:** Purchase intent action → Moderate-Strong

**Process:**
1. Present the product with pricing (multiple price tiers if testing price sensitivity)
2. Show a "Buy Now" button or pricing page
3. Track clicks, email sign-ups, and payment info entered
4. Stop before charging — inform the customer the product is not yet available
5. Follow up to maintain relationship

**Key metrics:**
- CTR on "Buy Now" button: Moderate evidence
- Email sign-up after price page: Moderate-Strong evidence
- Payment info entered (but not charged): Strong evidence

**Case study:** Joel Gascoigne tested Buffer by adding a "Plans and Pricing" page to his landing page before writing a single line of product code. $5/month tier generated the most email sign-ups, revealing customer willingness to pay and optimal price point.

### 13. Letter of Intent (LOI)

**Purpose:** A short, non-legally-binding written statement from a potential customer or partner confirming their intent to buy, partner, or collaborate. Ideal for B2B and institutional buyers.

**Evidence type:** Written commitment → Strong (stronger than verbal, weaker than payment)

**When to use:** B2B customer validation; key partner confirmation; institutional or government procurement planning.

**Process:**
1. Have initial conversations and confirm warm interest
2. Prepare a simple 1–2 page LOI template
3. Present the LOI during or after a meeting — never cold-send
4. Track: number sent, number signed, number leading to actual sale

**Sample LOI text:**
> We hereby submit a non-binding letter of intent to [purchase X units / partner on Y activity / participate in Z programme] under the following terms: [brief terms]. Signed: [Name, Title, Organisation, Date]

**Key insight from research:** When customers who said "I'd buy 1,000 units" were asked to write an LOI, they wrote "300 units." LOIs reveal real commitment vs. enthusiastic conversation.

**EA adaptation:** LOIs are widely used in Ugandan B2B and procurement contexts. Government agencies, NGOs, and large corporates often require LOIs before allocating supplier slots. Request LOIs early to qualify genuine buyers.

---

### 14. Pop-Up Store

**Purpose:** A temporary retail presence — a stall, stand, or rented space — used to test face-to-face customer interaction, value proposition messaging, and actual sales.

**Evidence type:** Real in-person interactions + sales → Strong

**When to use:** Physical product businesses; food and beverage; fashion; consumer goods. Not suitable for B2B.

**Process:**
1. Find a location with target customer traffic
2. Obtain required permits (KCCA trading permit; market authority permission)
3. Design the display and experience
4. Promote in advance (social media, WhatsApp, posters)
5. Staff with team members who can observe and record customer reactions
6. Collect: enquiries, email/phone contacts, mock sales, presales, actual sales

**Key metrics:** Number of visitors; enquiry rate; conversion to sale; average transaction value; qualitative themes from conversations

**EA adaptation:** Owino market, Nakasero market, Garden City atrium, Lugogo Mall pop-up spots, church trade fairs, school open days, and NEMA/UMA trade exhibitions are high-traffic, low-cost pop-up venue options in Kampala. Roadside stalls are the simplest form.

**Topology Eyewear case study:** After 2 hours at a San Francisco pop-up, they sold 4 pairs of glasses and learned that customers were "symptom aware but not problem aware" — they knew their glasses slid, but did not frame it as a "fit problem." This insight shaped years of marketing.

### 15. Extreme Programming Spike

**Purpose:** Write a small, time-boxed piece of code to test whether a specific technical approach is feasible — before committing to building the full system.

**Evidence type:** Technical proof → Strong (for feasibility)

**When to use:** When a business idea depends on a technical approach that has not been proven in your specific context. Spikes answer: "Can we actually build this?"

**Process:**
1. Define acceptance criteria: what must the spike prove?
2. Set a time box: 1 day to 2 weeks maximum
3. Write code focused only on the acceptance criteria
4. Share results with the team: performance, complexity, recommendation
5. Decide: build (custom), borrow (use existing library/service), or buy (use a third-party platform)
6. Discard the spike code — it is not production code

**Key output:** Build / Borrow / Buy recommendation with evidence

**EA adaptation:** Relevant for tech-enabled business ideas in Uganda: mobile money API integrations, USSD applications, SMS-based services, IoT sensor deployments. Spikes answer: "Can we reliably integrate with MTN API under Ugandan network conditions?"

## Choosing the Right Validation Experiment

| Situation | Recommended experiment |
|-----------|----------------------|
| Digital product, need to test flow | Clickable Prototype |
| Service business, want to validate before hiring | Concierge |
| Physical product, want to test sales | Pop-Up Store or Presale |
| B2B product, large deal size | Letter of Intent |
| B2C product, need proof of demand | Presale or Crowdfunding |
| Not sure if technical approach is possible | Extreme Programming Spike |
| Have traffic, want to optimise conversion | Split Test |
| Want to test price sensitivity | Mock Sale or Validation Survey |
| Tech product, want to test with stitched-together tools | Mash-Up |
| Want to test automation before building it | Wizard of Oz |

---

## Validation Experiment Progression

For most business ideas, follow this sequence:

1. **Low fidelity:** Clickable Prototype or Concierge (1–2 weeks; UGX 0–500,000)
2. **Medium fidelity:** Pop-Up Store, Presale, or Mock Sale (2–4 weeks; UGX 200,000–2,000,000)
3. **High fidelity:** Single Feature MVP or Mash-Up (4–12 weeks; UGX 1,000,000–10,000,000+)
4. **Commitment:** Letter of Intent or Crowdfunding (confirms full build is justified)

Only proceed to the next level when the current level provides at least **Somewhat Confident** evidence.
